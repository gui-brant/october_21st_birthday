ProsthetiX — Design Documentation

User scenarios · Activity diagrams · System state charts · Design decisions

Project Overview

v0.1 Draft

ProsthetiX is a project management web app tailored for October, a prosthetics engineer who struggles with starting projects. The app guides users from project creation through to task completion, with gamification (EXP/leveling), Google Calendar sync, and a structured task/subtask system.

This document reflects all design decisions resolved during the design review, including OTP limits, tap-out behavior, new-task insertion order, subtask display, timezone handling, EXP formula, Google Calendar authority model, trie empty states, and session persistence. See §Design Decisions for full detail.

Resolved Design Decisions

All resolved

D1 · OTP — Expiry & Failure State

Authentication

Max 5 OTP attempts allowed. After 5 failures the session is locked and the user is redirected to the login page with an error. A "Resend OTP" button is shown after 60 seconds. OTP expires after 5 minutes. The 5-attempt count and 5-minute window reset on each fresh OTP issuance.

D2 · New Project — Tap-Out Behavior

Project Creation

Input is trimmed of whitespace before evaluation. If trimmed result is empty, no project is created and the modal closes silently. If trimmed result is non-empty, the project is created immediately using that trimmed string as its title. Projects are deletable via a trash icon on the right of each project bar. Clicking trash shows an "Are you sure? [Yes] [Cancel]" confirmation modal.

D3 · New Task Insertion Order

Task Management

New tasks are inserted at the top (position #1) by default, bumping all existing tasks down by one. The numbering reflows dynamically. A future update will expose a Settings toggle to choose top vs. bottom insertion.

Future: Settings option — insert new tasks at top or bottom.

D4 · Subtask Display & Editing

Task / Subtask

Subtasks are revealed only when the parent Task is single-clicked (expanding a dropdown). The subtask list is indented within the Task's dropdown space. Double-clicking a task or subtask row enters inline edit mode for the name. Single-click on the row outside edit mode toggles expansion. Listing style (number/bullet/dash) is stored separately on Tasks and Subtasks in the data model even while the same control drives both — this prevents a schema migration when the UI is later split.

Future: Separate listing-style controls for Tasks vs. Subtasks.

D5 · Timezone Handling

Due Dates / Calendar

User's timezone is detected via Intl.DateTimeFormat().resolvedOptions().timeZone at signup and stored in their profile. All due date/time values are stored as UTC in the database and rendered in the user's stored timezone. The detected timezone is shown next to the Due Time input field and can be overridden in Settings (future).

D6 · EXP Bar

Gamification

EXP is only awarded when all tasks in a project are completed (ticked off). EXP gained = sum of estimated hours across all subtasks in the project × 100. Level-up thresholds are governed by the piecewise function E(L) = 100 × H(L). See §EXP Formula Reference.

D7 · Google Calendar Sync Authority

Calendar

Google Calendar is the source of truth for time (when events are scheduled). The app is the source of truth for task structure (what the event represents, its subtask linkage, completion state). If a user modifies or deletes an event in Google Calendar, the app reflects only the time change — the task itself is unaffected. Time conflicts are flagged in the app's Calendar tab with a warning icon.

D8 · Trie Empty State

Project Search

When the trie returns no matches, the project list area shows: "No projects found." in muted text. No auto-clear, no redirect. Whitespace is stripped from both the query and project names before trie traversal.

D9 · Session Persistence & Logout

Authentication

Default session timeout: 30 minutes of inactivity. Logging out clears the cookie immediately regardless of timeout setting. Future Settings will expose timeout options: 5 min, 15 min, 30 min (default), 1 hr, 3 hr, 1 day, 5 days, Never. "Never" means cookie persists until explicit logout only.

Future: Session timeout configurable in Settings.

D10 · Completed Task/Subtask Styling

Task / Subtask

Ticking a task or subtask darkens/greys the row background and applies strikethrough styling to the description text. If all subtasks under a task are ticked, the parent task is automatically ticked (and styled). Manual tick of the parent task is always available regardless of subtask state.

User Scenarios — Authentication

Auth

SC-A1 · First-time signup

Actor: New user (October)

October visits the app for the first time and creates a new account.

  1. Lands on Login page, clicks "Sign up"
  2. Fills: Name, Age, Email, Password, Retype Password
  3. All fields validated; passwords match → form submits
  4. OTP sent to her email; OTP entry screen appears
  5. October opens email, copies 6-digit OTP
  6. Enters OTP within 5 minutes → account created
  7. Redirected to Main tab with empty project list

SC-A2 · Returning login

Actor: Returning user

October returns to the app on a new day after her cookie has expired.

  1. Lands on Login page
  2. Enters email and password
  3. Credentials validated → OTP sent to email
  4. Enters OTP within 5 minutes → session cookie set
  5. Redirected to Main tab (projects intact)

SC-A3 · OTP failure & lockout

Actor: October (typo on OTP)

October misreads her OTP and keeps entering wrong codes.

  1. Enters wrong OTP — attempt 1 of 5, error shown
  2. Enters wrong OTP — attempts 2, 3, 4 — each shows remaining count
  3. Enters wrong OTP — 5th attempt → session locked
  4. Redirected to Login page with message: "Too many failed attempts. Please log in again."
  5. October re-enters email/password → new OTP issued, counter reset

SC-A4 · OTP expiry

Actor: Distracted user

October gets distracted before entering the OTP.

  1. Receives OTP email, gets sidetracked for 6 minutes
  2. Returns to OTP screen, enters (now expired) code
  3. System rejects: "OTP expired. Request a new one."
  4. Clicks "Resend OTP" (available after 60s from last send)
  5. New OTP sent; counter and 5-minute window reset

SC-A5 · Logout

Actor: October, shared computer

October is done working and wants to secure her session.

  1. Clicks "Log out" in top-right of Main tab
  2. Session cookie is immediately invalidated
  3. Redirected to Login page
  4. No cached project data accessible without re-auth

SC-A6 · Signup validation failure

Actor: New user

User attempts to sign up but leaves fields blank or mismatches passwords.

  1. Clicks "Sign up", partially fills the form
  2. Attempts to submit with missing fields → inline field errors shown
  3. Fills all fields but password ≠ retype → "Passwords do not match" error
  4. Corrects all fields → form proceeds to OTP step

User Scenarios — Project Management

Projects

SC-P1 · Create new project

Actor: October

October starts a new prosthetics project from the Main tab.

  1. Clicks "+ New Project" on Main tab
  2. Modal fades in: "What is the name of the project?" + large text box
  3. Types "Arm Mk3" and presses Enter
  4. Modal fades out; project title appears at top of project page
  5. Project appears in Main tab list (most recent, top)

SC-P2 · Tap out of new project (empty)

Actor: October (accidental click)

October accidentally clicks "+ New Project" but immediately clicks away.

  1. Modal appears with empty text field
  2. October clicks outside the modal
  3. Input trimmed → empty → no project created
  4. Modal closes silently, Main tab unchanged

SC-P3 · Tap out with partial title

Actor: October (incomplete but intentional)

October starts typing a name, gets interrupted, and clicks away.

  1. Modal open, October types "Arm"
  2. Clicks outside modal (or focus leaves)
  3. Trimmed input "Arm" is non-empty → project "Arm" created
  4. Project appears in list; title is editable by clicking it on the project page

SC-P4 · Search for a project

Actor: October (many projects)

October has 20 projects and wants to find "Arm 2" quickly.

  1. Clicks search box on Main tab
  2. Types "A" → trie returns all projects starting with A
  3. Types "Arm " → still shows "Arm 1", "Arm 2", "Arm Mk3" (whitespace stripped)
  4. Types "Arm 2" → only "Arm 2" shown
  5. Clicks "Arm 2" → navigates to that project

SC-P5 · Delete a project

Actor: October (obsolete project)

October wants to delete an outdated project.

  1. Sees project bar on Main tab; hovers → trash icon appears on right
  2. Clicks trash icon
  3. "Are you sure? [Yes] [Cancel]" modal appears
  4. Clicks "Yes" → project removed from list and database
  5. EXP previously earned from that project is retained

SC-P6 · Rename a project

Actor: October

October decides to rename a project after opening it.

  1. Opens project → sees title at top
  2. Clicks title → inline text input appears, title becomes editable
  3. Types new name, presses Enter → name saved permanently
  4. Navigates away without pressing Enter → auto-save triggers on blur

User Scenarios — Tasks & Subtasks

Tasks

SC-T1 · Add tasks to a project

Actor: October

October opens a fresh project and adds her first research tasks.

  1. Opens project → empty list, line bar with "New Task" button visible
  2. Clicks "New Task" → prompt changes to "Knowledge/Research needed…"
  3. Types "Research carbon fibre suppliers" → presses Enter → task inserted at #1
  4. Types "CAD model elbow joint" → Enter → becomes new #1, previous becomes #2
  5. Clicks "Done Adding Tasks" → "Are you sure? Yes / No" prompt appears
  6. Clicks "Yes" → returns to normal view with 2 tasks listed

SC-T2 · Reorder tasks by drag

Actor: October

October wants to reprioritise tasks after adding them.

  1. Sees task list; holds left side of task #3 row
  2. Drags upward — numbering on all rows updates live as she drags
  3. Releases above task #1 → formerly task #3 is now #1; others shift down
  4. Dragging works both inside and outside "Add Task" mode

SC-T3 · Add subtasks

Actor: October

October breaks down a task into subtasks with time estimates.

  1. Single-clicks task "Research carbon fibre suppliers" → dropdown opens
  2. Clicks "Edit or Add Subtask" inside dropdown
  3. Types "Email 3 suppliers for quotes" → Enter → subtask added at #1
  4. Enters estimated time: 0.5 hr; sets due date via calendar picker
  5. Clicks "Done Adding Subtasks" → confirms → subtask list shown indented

SC-T4 · Tick off subtask & task auto-completion

Actor: October

October finishes a subtask; the parent task auto-completes.

  1. Ticks checkbox on the only subtask under "Research carbon fibre suppliers"
  2. Subtask row turns grey, text strikes through
  3. All subtasks now ticked → parent task auto-ticked
  4. Parent task row also greys out and strikes through
  5. If all project tasks complete → EXP calculation triggers

SC-T5 · Change listing style

Actor: October (preference)

October prefers bullets over numbers for a particular project.

  1. In the project toolbar line, clicks the bullet (○) button
  2. All task prefixes update to bullets immediately
  3. Subtasks also update (same control drives both for now)
  4. Data model stores style separately for tasks vs. subtasks — enabling future split
Future: separate controls per level.

SC-T6 · Set subtask due date & calendar push

Actor: October

October assigns a deadline to a subtask and it appears in her calendar.

  1. In "Edit or Add Subtask" mode, clicks "None" next to Due Date
  2. Mini calendar picker opens; October selects a date
  3. Due Time field unlocks; October enters 3:00 PM (auto-timezone: EST)
  4. Saves → subtask has due date/time stored in UTC
  5. Event appears in app Calendar tab and is pushed to Google Calendar

User Scenarios — Calendar & Google Sync

Calendar

SC-C1 · View calendar tab

Actor: October

October switches to the Calendar tab to see all upcoming deadlines.

  1. Clicks Calendar button (top right of Main tab)
  2. FullCalendar renders with all subtask due dates as events
  3. Events are colour-coded by project
  4. October can click any event to jump to its subtask

SC-C2 · Google Calendar edit conflict

Actor: October (edits in Google Calendar)

October reschedules an event in Google Calendar; app reflects time update.

  1. October moves event in Google Calendar to a different time
  2. App syncs on next open (or periodic sync): event time updated
  3. Subtask due date/time updated to match new time
  4. Task structure, name, and completion state unchanged

SC-C3 · Google Calendar event deletion

Actor: October

October deletes a calendar event. The subtask due date is cleared.

  1. October deletes event in Google Calendar
  2. App syncs → subtask Due Date resets to "None", Due Time locked
  3. Subtask itself still exists and is not deleted
  4. Warning shown in Calendar tab: "1 subtask had its due date cleared via Google Calendar"

User Scenarios — EXP System

EXP

SC-E1 · Earn EXP completing a project

Actor: October

October completes all tasks in a project and gains EXP.

  1. All tasks in "Arm Mk3" ticked → project considered complete
  2. System sums all subtask estimated hours across the project: e.g., 6.5 hrs
  3. EXP gained = 6.5 × 100 = 650 EXP
  4. EXP bar in dashboard animates to new value
  5. If threshold crossed → level-up animation plays

SC-E2 · Level up across boundary

Actor: October (level 6 → 8)

October gains enough EXP to cross two level boundaries at once.

  1. Was at level 6 (needs 100 EXP/hr × 1 hr = 100 EXP to level)
  2. Earns 250 EXP from completed project
  3. Level 6 → 7 (100 EXP used), then 7 → 8 (200 EXP used)
  4. Level-up animation plays twice in sequence
  5. Remaining EXP (0) rolls into level 8 bar

SC-E3 · No EXP for partial project

Actor: October

October ticks 9 out of 10 tasks. EXP is NOT awarded yet.

  1. 9/10 tasks ticked in "Arm Mk3"
  2. EXP bar unchanged — no partial awards
  3. Final task ticked → all tasks complete → EXP triggers
EXP is all-or-nothing per project. No partial credit.

EXP Formula Reference

Gamification

EXP required to level up from level L is: E(L) = 100 × H(L), where H(L) is the hours required per reward at level L.

Level RangePhaseH(L) — hrs/rewardE(L) — EXP to next levelDuration rationale
1 – 7Week 11 hr100 EXP7 rewards × 1 hr/reward = Week 1
8 – 14Week 22 hrs200 EXP7 rewards × 2 hr/reward = Week 2
15 – 21Week 33 hrs300 EXP7 rewards × 3 hr/reward = Week 3
22 – 28Week 44 hrs400 EXP7 rewards × 4 hr/reward = Week 4
29 – 756Years 1–2400 + (400/728)(L − 28) hrsScales 400→800 EXP2 years at 7 rewards/week: 4 hr → 8 hr
757 – 1848Years 3–5800 + (800/1092)(L − 756) hrsScales 800→1600 EXP3 years at 7 rewards/week: 8 hr → 16 hr
1849+Endgame1600 hrs (cap)160,000 EXPCap achieved at level 1849; scales infinitely
EXP is only awarded when ALL tasks in a project are complete. Partial completion awards nothing. EXP earned from a deleted project is retained by the user.

Planned Update Notes

Roadmap
Task insertion order: Default is top (position #1). A future Settings toggle will allow choosing top vs. bottom insertion for new tasks and subtasks.
Listing style (number / bullet / dash): Currently one control drives both Tasks and Subtasks. The data model stores style separately. A future update will expose separate controls per level.
Session timeout: Default is 30 minutes. A future Settings panel will expose options: 5 min, 15 min, 30 min, 1 hr, 3 hr, 1 day, 5 days, Never. "Never" = persist cookie until explicit logout.
User Settings (future patch): View and modify name and age. View email address. Option to change app layout. Session timeout configuration. Active sessions list.
Subtask listing style independence: Once the Settings panel is introduced, Task listing style and Subtask listing style will be configurable separately. Schema already supports this (stored independently).
Google Calendar two-way sync: Currently the app pushes due dates to Google Calendar and respects time edits/deletions from Google (Google = source of truth for time). Full bidirectional conflict resolution UI is a future milestone.
Timezone override: Timezone is auto-detected on signup. A future Settings option will allow the user to override their stored timezone.
